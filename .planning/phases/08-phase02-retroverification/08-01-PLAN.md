### DARM-02 — NVR pre-check before disarm
Full narrative: Before any ISAPI snapshot calls, service.py:[lines] iterates all NVRs referenced by partition cameras and calls get_device_info() for each. On exception (timeout, connection error), sets state.state = "error", writes error_detail, commits, and returns HTTP error. Partial success (some endpoints fail) is allowed per spec ("saves only endpoints returning HTTP 200"). test_disarm_success exercises all four types fetched successfully.

### DARM-04 — Snapshot immutability (do not overwrite existing snapshot)
Full narrative: This is the most complex requirement in Phase 02. Implementation uses two sequential queries at service.py:[lines]:

Query 1 (partition-scoped): checks if THIS partition already has a snapshot for this camera. If found, the entire snapshot block is skipped — no new ISAPI GETs, no new snapshot written. This handles idempotent re-disarm.

Query 2 (camera-scoped ANY): if no partition-scoped snapshot found, the code fetches live detection config from ISAPI, then checks whether ANY other partition has a snapshot for this camera (service.py:[line], no WHERE on partition_id). If existing_any is found, final_snapshot_data = existing_any.snapshot_data — the live ISAPI data is DISCARDED in favor of the existing snapshot. This preserves the original armed state when a second partition disarms a camera that is already disarmed.

Two tests exercise this: test_disarm_snapshot_protection pre-seeds a snapshot for the SAME partition and verifies the existing data is unchanged after a second disarm call. test_disarm_camera_already_disarmed_by_other_partition seeds a snapshot from a DIFFERENT partition and verifies the second disarm uses that snapshot data rather than fresh ISAPI data.

### DARM-05 — Disable enabled detections via PUT
Full narrative: service.py:[lines] iterates snapshot_data items. For each detection type, _is_enabled_in_xml(xml) at service.py:[line] parses the XML and checks for enabled=true in the first matching element. Only if true does the code call _disable_in_xml(xml) at service.py:[line], which replaces the enabled element's text with "false" while re-serializing the complete XML tree (preserving all other settings), then calls client.put_detection_config(). Detection types that were already disabled in the snapshot are silently skipped. test_disarm_success verifies put_detection_config is called for enabled detections.

### DARM-10 — Parallel ISAPI calls (SATISFIED BY INSPECTION)
Status: SATISFIED (by inspection) — structural behavior; no automated concurrency test.

Narrative: Per the evidence standard for structural/async behaviors, this requirement is satisfied by code inspection. service.py:[line] defines db_lock = asyncio.Lock() to protect concurrent DB writes during parallel camera processing. service.py:[line] executes await asyncio.gather(*(process_camera(c) for c in cameras)) — this spawns all per-camera coroutines concurrently rather than awaiting them sequentially. The same pattern is used in arm_partition at service.py:[line]. The asyncio.Lock() ensures that DB operations inside process_camera that write to the session (snapshot creation, refcount update) are serialized, while the ISAPI HTTP calls (which do not touch the session) run in true parallel. STATE.md records this design decision: "asyncio.Lock used in disarm_partition loop to prevent concurrent DB writes on the same AsyncSession from causing SAWarnings during parallel ISAPI calls."
